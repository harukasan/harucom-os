## 6. 640x480px stable output

まず最初のテーマとして、640x480ピクセルの安定した映像出力をどう実現しているかを見ていきます。

## 7. Pixel clock frequency

デジタルビデオを出力するためには、ピクセルのストリームを絶え間なく送り続ける必要があります。

640x480pxの場合はブランクエリアも合わせると800x525px、これを60Hzなので25.2MHzになります。つまり40nsに1ピクセル送らないといけない計算です。

## 8. TMDS Signal

DVIはTMDS伝送を用いています。これは8bitを10bitにエンコードし、RGBそれぞれの差動信号として送るものです。つまり、各チャンネルは25MHz x 10bit、250 Mbpsでおくる必要があります。

## 9. HSTX

これをRP2350で実現するための機能がHSTXです。
これはGPIOの12-19ピンにだけついている機能でTMDSのハードウェアエンコード機能があります。
またDMAでデータ供給できるのでCPUを介在せずにTMDS伝送を実現します。

## 10. How HSTX uses

HSTXにはコマンドエクスパンダという機能があり、FIFOに流すデータにコマンドを混ぜることで動作を制御します。

コマンドは2種類、固定データを繰り返すRAWと、ピクセルデータをTMDSエンコードするTMDSです。
HSync区間はRAWコマンドで同期信号を繰り返し、アクティブ領域はTMDSコマンドでピクセルをエンコードして出力します。
これにより、DMAでコマンドとピクセルデータを流し込むだけで1スキャンラインのTMDS信号が生成されます。

## 11. Data rate for DMA

HSTXをつかうことでTMDSエンコードはできるようになりました。
つぎの問題はDMAです。

Harucomでは内部の色フォーマットにRGB332、つまり1pixelあたり8bitのデータを使っています。つまり必要なビットレートは25MHz x 8bit、200Mbpsです。

1スキャンライン、つまり800pxだと32usです。
これをHSTXのFIFOに対して送り続ける必要があります。

## 12. Multi-core

ここでHarucomのCPU構成について触れておきます。

RP2350には2つのCPUコアが搭載されています。HarucomではCore 0をRuby VMの実行に、Core 1をDVI出力に割り当てています。

Core 1はDMAにピクセルデータを供給し続ける役割を持ちます。この役割分担が、これから説明する2つのディスプレイモードの設計に大きく影響しています。

## 13. Two display modes

ここまではDVI出力の共通基盤の話でした。ここからは、画面のピクセルデータをどうつくるか、という話に入ります。

Harucomには2つのディスプレイモードがあります。ひとつは先程IRBの画面でつかっていたテキストモード、そしてもうひとつはいま使っているグラフィックスモードです。

どちらのモードも640x480pxの解像度を持っていますが、画面出力方法は異なります。

## 14. Graphics mode

まずグラフィックスモードから見てみましょう。こちらはシンプルです。

グラフィックスモードでは640x480ピクセルのフレームバッファをSRAM上に持っています。RGB332なので1ピクセル1バイト、約307KBです。
DMAはこのフレームバッファを直接読み出してHSTXに流すだけなので、CPUがスキャンラインごとになにかする必要はありません。

## 15. Text mode: why scanline rendering?

つぎにテキストモードです。こちらは少し複雑です。

テキストモードではフレームバッファを持ちません。かわりに106桁x37行のテキストVRAMを持っています。1セルあたり4バイト、全体で約15KBです。グラフィックスモードの307KBと比べると大幅にメモリを節約できます。

しかしDMAに流すのはピクセルデータでなければなりません。そこでスキャンラインごとにVRAMとフォントデータからピクセルを生成する、スキャンラインレンダリングを行います。

つまりテキストモードでは、DMAがピクセルを送出している裏で、CPUが次のラインをレンダリングし続ける必要があるのです。

## 16. Graphics mode vs Text mode

2つのモードを比べてみましょう。

グラフィックスモードはフレームバッファをDMAが直接読むだけなので、Core 1にレンダリングの負荷がかかりません。そのかわり、640x480で約307KBのSRAMを消費します。ダブルバッファリングにはPSRAM上にもう1面必要です。

テキストモードはVRAMが約15KBと省メモリです。しかしCore 1がスキャンラインごとにフォントからピクセルを生成しなければならず、厳しいサイクル予算の中でレンダリングを行う必要があります。

IRBやエディタのようなテキストベースのアプリケーションには、メモリ効率のよいテキストモードが適しています。一方、P5のような描画ライブラリにはグラフィックスモードが必要です。

では、テキストモードのスキャンラインレンダリングをどう実現しているか、もう少し詳しく見ていきましょう。

## 17. System clock

ここで問題になるのがサイクル予算です。

RP2350の標準システムクロックは150MHzです。
これだと32usで使えるのは4800サイクルくらいになりますが、
実際にはDMAがメモリのバスを占有している間は使えないため、動作できるのはHブランキングの時間だけ、つまり1000サイクルほどしかありません。

たった1000サイクルほどで640pxのスキャンラインをレンダリングするのは無理があります。

最初は150MHzでがんばろうとしていたですが、この問題を解決するためRP2350を250MHzにオーバークロックすることにしました。

オーバークロックした場合、ブランキング中に1600サイクルほどつかえる他、HSTXが125MHzで動いているのに対して、CPUは250MHzで動くので2:1の比率でCPUがRAMにアクセスできるようになります。そうするとレンダリングに使える時間が2200サイクルぐらいになります。

## 18. Batch rendering

オーバークロックしても1スキャンラインあたり2200サイクルではまだ余裕がありません。そこでバッチレンダリングを導入しました。

1ラインずつレンダリングするのではなく、4ライン分をまとめてレンダリングしてからDMAに渡します。
DMAが前の4ラインを送出している間に、CPUは次の4ラインをレンダリングします。これにより使えるサイクル予算が約32,000サイクルに広がり、十分な余裕が生まれます。

これを実現するために、ラインバッファを8本用意しています。DMAが4本を読んでいる間にCPU���残りの4本に書き込む、ダブルバッファ方式です。

## 19. CJK wide character support

テキストモードで日本語を表示するためにはもうひとつ工夫が必要です。

半角文字は6px幅で256種類しかないので、起動時にすべてSRAM上のキャッシュに読み込んでおけます。しかし全角文字はJIS X 0208だけで8,000字以上あり、12px幅で1文字あたり2バイト x 13スキャンラインのデータが必要です。すべてをSRAMに載せることはできません。

そこでオンデマンドのワイドグリフキャッシュを導入しています。512スロットのキャッシュをSRAM上に確保し、1024エントリのオープンアドレスハッシュテーブルで管理します。

全角文字がVRAMに書き込まれるとき、Core 0がフラッシュからグリフデータを読み出してキャッシュに格納します。VRAMのセルにはJISコードではなくキャッシュスロット番号を保存します。こうすることで、Core 1のレンダリング中にフラッシュへのアクセスが一切発生しません。

512スロットを使い切った場合は、VRAMの全セルをスキャンして実際に使われているグリフだけを収集し、キャッシュを再構築します。画面上には最大1,961文字分の全角文字を表示できますが、通常のテキストでは同じ文字が繰り返し現れるため、ユニークな種類数は512に収まることがほとんどです。

## 20. Bus Contention

ここまでの最適化で安定したDVI出力を実現できました。

長かったですね、じゃあRuby VMを起動しよう、とおもったらまたDVI信号が止まってしまいました。

なにが、おこっていたのか。バス競合です。


## 21. Bus Contention Patterns

バス競合は2箇所で起きます。

ひとつめはSRAMバスです。RP2350のSRAMは10バンクに分かれていて、異なるバンクなら同時アクセスできますが、同じバンクに対してCore 0、Core 1、DMAが同時にアクセスできません。DVI出力中はCore 1のレンダラとDMAがSRAMを使い続けているので、Core 0のRuby VMがSRAMにアクセスすると競合します。

ふたつめはQMIバスです。フラッシュとPSRAMは同じQMIコントローラを共有しており、Core 0がフラッシュからバイトコードを読むときにXIPキャッシュミスが発生すると、QMIへのアクセスが必要になり、ここでも競合が起きます。


## 22. Flash Access Mitigations

バス競合の対策として、まずCore 1がフラッシュにアクセスしないようにしました。

レンダリング関数とDMA IRQハンドラはSCRATCH_Xに配置して、命令フェッチでフラッシュにアクセスしないようにしています。
フォントデータもSRAM上にキャッシュを構築して、レンダリング中のフラッシュ読み出しを完全に排除しました。
Core 1のベクタテーブルもSRAMにコピーしています。

これでCore 1はフラッシュに一切アクセスしなくなり、QMIバスの競合がなくなります。

## 22.5. SCRATCH_X / SCRATCH_Y

RP2350にはメインSRAMとは別にSCRATCH_XとSCRATCH_Yという2つの専用SRAMバンクがあります。これらはメインSRAMとは独立したバスポートを持っているので、メインSRAMへのアクセスと同時にアクセスしても競合しません。

デフォルトではここにスタックが置かれていますが、スタックを追い出してDVI出力のために活用しています。
Core 0のスタックはメインSRAMのBSS領域に32KBを確保し、Core 1のスタックも別途メインSRAMに確保しました。

空いたSCRATCH_Xにはレンダリング関数とDMA IRQハンドラのコードを配置しています。命令フェッチがメインSRAMやフラッシュと競合しません。
SCRATCH_Yにはフォントのニブルマスクテーブルを配置しています。レンダリングのタイトループ中のルックアップがメインSRAMのDMAと競合しません。

## 23. Bus Priority

つぎにSRAMバスの競合対策です。

RP2350のバスファブリックにはプライオリティの設定があります。DMAに高い優先度を設定することで、SRAMバンクの競合が発生したときにDMAが優先的にアクセスできるようにしています。
これによりHSTXのFIFOが枯渇して映像が途切れることを防ぎます。

## 24. Line Buffer Bank Striping

もうひとつSRAMバスの対策として、ラインバッファのメモリ配置を工夫しています。

RP2350のSRAMは8バンクに分かれていて、ワード単位でバンクが割り当てられます。
ラインバッファのストライドを644バイト、つまり161ワードにしています。161は8と互いに素なので、8本のバッファがすべて異なるバンクに配置されます。
DMAとCPUが同じピクセル位置にアクセスしても、バッファが違えばバンクが違うので競合しません。


## 25. Boot Sequence

ここからはソフトウェア側、Rubyの世界の話に移ります。

電源が入るとまずCore 0でフラッシュ上のFATファイルシステムをマウントし、PSRAMを初期化してmrubyのヒープとして使えるようにします。
そしてCore 1を起動してDVI出力を開始します。
最後にCore 0でmruby VMを起動し、ファイルシステム上の/system.rbをロードします。

## 26. system.rb

system.rbはHarucom OSのエントリポイントです。

ここでUSBホストのポーリングタスクとキーボード入力タスクをバックグラウンドタスクとして起動します。
そしてコンソール、キーボード、ラインエディタを初期化して、IRBを起動します。

## 27. Multi-tasking in mruby VM

Harucom OSではPicoRubyのTaskクラスを使った協調的マルチタスクを実装しています。

1msごとにタイマー割り込みが発生してスケジューラが動き、Task.passで明示的にCPUを譲ります。
これによりUSBポーリングやキーボード入力をバックグラウンドで処理しながら、メインのRubyスクリプトを実行できます。

## 28. IRB

起動するとIRBが立ち上がります。

入力はラインエディタによる複数行編集に対応しており、構文が完結するまで継続行として入力を受け付けます。
入力されたコードはSandboxの中でコンパイル、実行されます。

## 29. Reline-like Multiline Editor

ラインエディタはRubyのRelineライブラリに近い機能を持っています。

カーソル移動、行の挿入削除、スクロールバック、UTF-8の全角文字幅対応などを、すべてRubyで実装しています。
DVI::Textのテキストモードの上にコンソールクラスを重ね、ANSIエスケープシーケンスによる色表示にも対応しています。

## 30. Sandbox in PicoRuby

ユーザーのコードはSandboxの中で実行されます。

Sandboxは隔離されたmruby VMインスタンスで、コンパイルと実行を分離しています。
タイムアウト付きで実行でき、暴走したスクリプトを安全に停止できます。
フラッシュ上の.mrbファイルはメモリにコピーせず直接実行することで、メモリを節約しています。

## 31. P5

最後にグラフィックスAPIの紹介です。

P5はProcessing/p5.jsにインスパイアされた描画ライブラリで、Rubyから簡単にグラフィックスを描けます。
rect、circle、line、textといった基本図形のほか、アフィン変換やブレンドモードにも対応しています。

状態管理はRuby側、描画処理はC側という分担で、640x480のフレームバッファに対してダブルバッファリングで描画します。
