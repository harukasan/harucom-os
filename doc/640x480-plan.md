# 640x480 text mode: remaining work

## Current status: wide character FIFO underflow analysis

Full-screen CJK stress test (37 rows x 53 wide chars, rewritten every
frame from mruby) shows render_last ~2,250-2,350 cycles with occasional
fifo_min=0 events (fifo_empty increments ~1 per 1000 frames). The system
is functional but not fully stable under worst-case load.

### Observed data

```
render_last:  2,250-2,350 cycles (all rows wide + mixed attr)
irq_max:      2,560 cycles
fifo_min:     1 (typical), 0 (rare, ~1 per 1000 frames)
fifo_empty:   2 events in 2,131 frames
underflow lines: 87, 32 (no pattern -- varies per run)
s9c (SCRATCH_Y contested): 0 (no contention on expanded_nibble)
```

### Timing breakdown

The blanking budget is 1,600 cycles. The FIFO provides 320 cycles of
buffer (8 entries x 40 cycles). Total budget before underflow: 1,920
cycles. The measured render_last of ~2,350 exceeds this by ~430 cycles.

Why does it mostly work? The render_last measurement includes the full
IRQ overhead (diagnostics, descriptor build). The actual pixel data
reaches the FIFO during the active region too, since the DMA descriptor
was triggered before the render started. The render only needs to stay
ahead of the FIFO drain rate: 1 word consumed every 40 cycles. With an
8-word FIFO starting full, the render has blanking + 320 cycles before
the first underflow.

The rare fifo_empty=0 events likely occur when render timing jitter
(bus arbitration, cache miss) pushes a single scanline past the limit.

### Debugging approach

To identify the exact cause of the rare underflows:

1. **Per-scanline render timing histogram**: record render cycles in a
   256-entry histogram (bucket = cycles / 16). Print the histogram
   every N seconds. This reveals whether underflows come from a long
   tail or a consistent near-budget render.

2. **Log the row type on underflow**: when fifo_empty fires, record
   `text_row`, `row_has_wide[text_row]`, and `row_uniform_attr[text_row]`
   alongside the scanline number. This identifies whether underflows
   correlate with specific row properties (all-wide, mixed-attr, etc.).

3. **Core 0 write interference**: Core 0 writes to text_vram (main SRAM)
   every frame. If a write collides with the render's cell load on the
   same SRAM bank, it adds 1 wait cycle. To test: add a `__dmb()` in
   the Ruby loop before `wait_vsync` to batch writes, or temporarily
   disable mruby to see if underflows vanish.

4. **DMA descriptor timing**: the CMD DMA reads the 12-word descriptor
   buffer from main SRAM. If this burst overlaps with the render's first
   few cell loads, it can steal cycles. To test: move
   `dma_scanline_buf` to SCRATCH_Y (requires freeing the stack_dummy).

### Candidate fixes (ordered by impact / effort)

**1. Disable diagnostics** (immediate, ~280 cycle saving):
Remove `DVI_DIAGNOSTICS` from CMakeLists.txt. Drops IRQ overhead from
~280 to ~50 cycles. Expected render_last: ~2,070. Still above 1,920 but
with more margin. May eliminate the rare underflows entirely.

**2. H_BACK_PORCH increase** (1 line change, +320 cycle budget):
Increase back porch from 48 to 80 pixels. Total budget becomes 1,920 +
320 = 2,240 cycles. At render_last ~2,070 (diagnostics off), this gives
170 cycles margin. Frame rate drops to ~57.9 Hz.

**3. Attr run batching** (medium effort, ~200-400 cycle saving):
Pre-scan each text row at row boundaries (every 13 scanlines) to build
a run-length list of attr regions. The render loop processes runs instead
of per-cell attr checks. For uniform-color CJK rows (common case), this
approaches the uniform-attr fast path speed.

**4. VBlank pre-rendering** (large effort, eliminates the problem):
Render dirty rows during VBlank (360,000 cycles). The IRQ handler serves
pre-rendered scanlines from a cache. Rows that change between frames are
re-rendered in the next VBlank. This completely decouples render timing
from the blanking budget. See architecture section below.

**5. Free SCRATCH_Y stack_dummy** (small effort, enables other opts):
Set `PICO_STACK_SIZE=0` to reclaim 2 KB. Move `dma_scanline_buf` or
`text_palette32` there to reduce main SRAM bus contention during render.
Risk: boot sequence uses this stack before main() switches to BSS stack.
Needs testing.

## VBlank pre-rendering architecture

Move scanline rendering from the per-scanline IRQ to the vertical blanking
period (360,000 cycles/frame). Pre-render all dirty rows into a per-row
scanline cache. The IRQ handler points DMA at the cached data instead of
rendering inline.

Scanline cache per text row: 13 scanlines x 640 bytes = 8,320 bytes.
Full screen (37 rows): 307 KB -- too large for SRAM. Practical options:

- **Partial cache**: allocate 3-4 row buffers (~25-33 KB). Typical console
  changes 1-2 rows per frame. LRU or dirty-priority eviction.
- **Ring cache**: keep a sliding window of recently rendered rows. Rows
  scrolled off screen are evicted.

Benefits:
- Eliminates rendering from the IRQ handler entirely for cached rows
- Zero bus contention during active pixels (DMA reads cache, CPU idle)
- Mixed-attr and wide-char rows are no longer timing-critical
- VBlank provides 360,000 contention-free cycles for rendering

Implementation: on VBlank (frame_count increment in IRQ), signal Core 1's
main loop (WFE) to render dirty rows into the cache. The IRQ handler checks
cache validity per text row and falls back to inline rendering on miss.

Note: Core 1 must not execute flash-resident code after BASEPRI is set.
The pre-render function must be placed in SRAM via `__not_in_flash_func()`
or `__scratch_x()`. During VBlank, flash data reads (font cache population)
are acceptable since no DMA HSTX reads are active, but code execution from
flash would be blocked by BASEPRI if an interrupt fires.

## Completed: SRAM wide glyph cache

The SRAM glyph cache approach has been implemented:

- 512-slot row-major cache in SRAM (26 KB regular + bold)
- Hash table (1024 entries) maps linear JIS index to cache slot
- Cache populated on Core 0 during `dvi_text_put_wide_char()` (no flash
  access on Core 1)
- Render path uses slot-indexed SRAM ldrh (no multiply, no flash access)
- All nibble mask lookups via expanded_nibble in SCRATCH_Y
- Cache rebuild on overflow scans VRAM and re-maps slots in-place

Result: render_last ~2,250-2,350 cycles for full-screen CJK (was 20,000+
with flash XIP contention). fifo_empty=0 for most runs. Stable under
full-screen CJK stress test with mruby active.

## Completed: conditional diagnostics

IRQ handler diagnostics (FIFO stat, DWT cycle counting, bus counters) are
now guarded by `#ifdef DVI_DIAGNOSTICS` (defined in CMakeLists.txt).
Removing the definition saves ~200-280 cycles per scanline.

## Completed: 250 MHz stabilization

The sys_clk reduction from 375 MHz to 250 MHz has been completed. The
render now fits within the blanking budget (1,600 cycles) via three
optimizations:

- Pre-expanded nibble table (2 KB) in SCRATCH_Y (zero DMA contention)
- Uniform-attribute fast path (skips per-cell attr checks)
- DMA trigger reordering in IRQ handler

Result: render_last=1,763, fifo_empty=0, fifo_min=7 at 250 MHz.
Stable over 3000+ frames at 250 MHz / VREG 1.15V. Standard 60 Hz timing
maintained. See [doc/dvi.md](dvi.md) for details.
