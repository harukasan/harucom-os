# 640x480 DVI output implementation plan

## Background: lessons from 720p

720p (1280x720 @ 60 Hz) output was implemented and is functional for light
workloads. However, heavy mruby VM execution destabilizes the DVI signal.
This document summarizes the root causes found during 720p debugging and
outlines the plan for stable 640x480 output.

### Root cause: QMI CS0/CS1 bus contention

The RP2350's QMI bus is shared between flash (CS0) and PSRAM (CS1). The
16 KB XIP cache covers both. When the mruby VM alternates between flash
code (C functions, Kernel#loop) and PSRAM data (bytecode, heap), the XIP
cache thrashes between CS0 and CS1, generating heavy QMI traffic.

The HSTX FIFO is only 8 entries deep. At 720p pixel rate (74.4 MHz), this
provides approximately 860 ns of buffer. Heavy QMI traffic causes enough
bus fabric disruption to delay DMA transfers beyond this buffer, causing
HSTX FIFO underflow and DVI signal loss.

### What works at 720p

| Configuration | Result |
|---|---|
| Core 1 BASEPRI isolation (DMA IRQ in SCRATCH_X) | Effective |
| SEV/WFE cross-core vsync signaling | Correct |
| `while true` main loop (single irep, no yield) | Stable |
| Light mruby (`loop do sleep_ms(500) end`) | Stable at VREG 1.40V |
| DMA + PROC1 high bus priority | Partial improvement |
| VREG 1.50V | Partial improvement |

### What fails at 720p

| Configuration | Result | Cause |
|---|---|---|
| `loop do` with `while` body | Signal lost | XIP cache thrash: flash Kernel#loop irep and PSRAM block irep alternate rapidly |
| `DVI.set_pixel` x1000/frame | Signal lost | Each C method call switches flash (C code) and PSRAM (bytecode) |
| `DVI.fill` every frame (230 KB memset) | Signal lost | Massive SRAM write bandwidth saturates bus fabric |
| vm.o in SRAM (.time_critical) | Signal lost | SRAM contention between DMA pixel reads and Core 0 VM code execution |

### Pixel buffer approach (attempted, insufficient for 720p)

Concept: copy each scanline from framebuf (main SRAM) to a dedicated
buffer in SCRATCH_X or SCRATCH_Y. DMA reads from the dedicated buffer
instead of main SRAM, isolating the HSTX data path from Core 0's bus
traffic.

Results:

| pixel_buf location | DVI stable? | Notes |
|---|---|---|
| Main SRAM | No | Same bus as Core 0, no isolation benefit |
| SCRATCH_X | No | 3-way contention: Core 1 I-fetch + Core 1 D-write + DMA D-read |
| SCRATCH_Y (no mruby) | Yes, FIFO empty = 0 | Effective: only 2-way contention (D-write + DMA D-read) |
| SCRATCH_Y (with mruby) | No | Core 1 memcpy reads framebuf from main SRAM; delayed by Core 0 QMI traffic |

The Core 1 software memcpy (160 word burst from SRAM) is worse than DMA's
hardware-paced reads (DREQ-spread, one word at a time). The concentrated
burst from the memcpy stalls more than the distributed DMA accesses.

A DMA-based async copy (using a third DMA channel) might work but was not
tested.

### SCRATCH_Y stack overlap

The default Core 0 stack lives in the top 2 KB of SCRATCH_Y
(`StackSize` = `PICO_STACK_SIZE` = 0x800, defined in `crt0.S`). Adding
pixel_buf (1280 B) to SCRATCH_Y leaves only ~640 B for the stack, causing
the mruby compiler to crash.

The `--defsym=__STACK_SIZE=0x8000` linker option does NOT work because the
stack size is actually controlled by the C preprocessor macro
`PICO_STACK_SIZE` in `crt0.S`, not the linker symbol `__STACK_SIZE`.

A BSS stack switch (MSP trampoline in `main()`) was tested and functions
correctly, but adds main SRAM bus traffic that contributes to DVI
instability at 720p.

### Key hardware constraints at 720p

- HSTX FIFO buffer: ~860 ns (8 entries at 74.4 MHz pixel clock)
- QMI transaction time: ~300 ns per PSRAM access
- Scanline period: 22.2 us (1650 pixels / 74.4 MHz)
- sys_clk: 372 MHz (3x overclock, VREG 1.35-1.50 V)

The margin between FIFO buffer (860 ns) and QMI transaction time (300 ns)
is too tight for reliable operation under heavy mruby load.

## 640x480 implementation plan

### Why 640x480

640x480 @ 60 Hz (standard VGA) has a pixel clock of 25.175 MHz, roughly
1/3 of 720p. This provides:

| Parameter | 720p | 640x480 | Improvement |
|---|---|---|---|
| Pixel clock | 74.4 MHz | 25.0 MHz | 3x slower |
| HSTX FIFO buffer | ~860 ns | ~2.56 us | 3x more margin |
| Scanline period | 22.2 us | 31.8 us | 1.4x longer |
| sys_clk required | 372 MHz | 125 MHz | No overclock |
| VREG | 1.35-1.50 V | 1.10 V (default) | No voltage risk |
| Framebuffer (320x240 2x) | 230 KB | 76.8 KB | 3x smaller |

The 3x larger FIFO buffer is the critical improvement. At 2.56 us buffer
time, even heavy QMI traffic (multiple 300 ns transactions) cannot drain
the FIFO.

### System clock

The system clock can remain at the default 125 MHz (or 126 MHz for an
exact 25.2 MHz pixel clock):

```
125 MHz / 5 = 25.0 MHz pixel clock (-0.7% from 25.175 MHz, within tolerance)
126 MHz / 5 = 25.2 MHz pixel clock (+0.1% from 25.175 MHz, ideal)
```

PLL configuration for 126 MHz:
- VCO = 12 MHz x 63 = 756 MHz (within 750-1600 MHz range)
- Post-dividers: /6/1 = 126 MHz

Or simply use the default 125 MHz and accept the -0.7% deviation.

At 125 MHz, no overclocking is required:
- VREG stays at default 1.10 V
- No `vreg_disable_voltage_limit()` needed
- Flash CLKDIV can remain at default 2 (SCK = 62.5 MHz, within 133 MHz spec)
- PSRAM CLKDIV = 2 (SCK = 62.5 MHz, within 109 MHz spec)

### HSTX configuration

```
clk_hstx     = clk_sys = 125 MHz
CLKDIV       = 5
N_SHIFTS     = 5
SHIFT        = 2 bits
Bit clock    = 125 / 5 x 5 x 2 = 250 Mbps
Pixel clock  = 125 / 5 = 25.0 MHz
```

TMDS encoding and shift expander configuration remain the same as 720p
(RGB332, 2x horizontal scaling).

### 640x480 timing (DMT/VGA)

| Parameter | Pixels/Lines |
|---|---|
| H active | 640 |
| H front porch | 16 |
| H sync width | 96 |
| H back porch | 48 |
| **H total** | **800** |
| V active | 480 |
| V front porch | 10 |
| V sync width | 2 |
| V back porch | 33 |
| **V total** | **525** |

Sync polarity: HSYNC negative, VSYNC negative (unlike 720p which is
positive/positive). The sync command tables must be updated.

### Framebuffer

Two options:

**Option A: 320x240 with 2x scaling (recommended)**
- Framebuffer: 320 x 240 x 1 byte (RGB332) = 76,800 bytes (75 KB)
- Horizontal 2x: HSTX shift expander (same as current)
- Vertical 2x: DMA IRQ handler maps 2 output lines per framebuffer row
- Output: 640x480 from 320x240 source
- Advantage: 3x less SRAM, 3x less DMA bandwidth, larger pixels for text

**Option B: 640x480 direct**
- Framebuffer: 640 x 480 x 1 byte (RGB332) = 307,200 bytes (300 KB)
- No horizontal scaling
- Advantage: full pixel resolution
- Disadvantage: large framebuffer, more DMA traffic

Option A is recommended. 75 KB framebuffer leaves more SRAM for stacks
and C data. The mruby heap is on PSRAM and unaffected.

### Core assignment (unchanged)

```
Core 0: mruby VM, stdio, timers
Core 1: DVI output (HSTX + DMA), BASEPRI-isolated
```

BASEPRI on Core 1 should still be used even at 640x480. While the margins
are much larger, BASEPRI prevents unexpected flash-resident ISRs from ever
running on the DVI core, providing defense in depth.

### DMA architecture (unchanged)

The CMD-to-DATA per-scanline DMA architecture remains the same. The DMA
reads directly from the framebuffer in main SRAM (no pixel_buf needed at
640x480 due to the larger FIFO margin).

If testing reveals instability even at 640x480 (unlikely), the DMA-based
async copy approach can be revisited: use a third DMA channel to copy
scanlines from framebuf to a SCRATCH_Y buffer, triggered by the IRQ
handler.

### Stack

The `PICO_STACK_SIZE` issue should be resolved:
- Set `PICO_STACK_SIZE` via `add_compile_definitions(PICO_STACK_SIZE=0x800)`
  in CMakeLists.txt (explicit, matching the default)
- If the mruby compiler needs more stack, implement the BSS stack switch
  trampoline. At 125 MHz without overclocking, the bus contention from BSS
  stack access is much lower and should not affect DVI.

### mruby DVI API

The DVI class methods (set_pixel, get_pixel, fill, fill_rect, wait_vsync,
frame_count, WIDTH, HEIGHT) should be re-added. The presym table must
include `fill_rect` (run `rake distclean` to regenerate).

### Implementation steps

1. Create new timing constants for 640x480 in `dvi_output.c`
2. Update `dvi_clock.c`: remove overclocking (use 125 MHz default) or
   configure PLL for 126 MHz
3. Update `dvi_output.c`: framebuffer size (320x240), sync polarity
   (negative/negative), sync command tables
4. Update `dvi_output.h`: new WIDTH/HEIGHT defines
5. Update `src/main.c`: remove vm_to_ram, adjust demo code
6. Update `mrbgems/picoruby-dvi/src/mruby/dvi.c`: DVI class API
7. Update `CMakeLists.txt`: remove vm_to_ram script, fix stack size
8. Test with heavy mruby workloads (`loop do` + methods + PSRAM)

### Test criteria

- `loop do` with complex body: stable
- `def` method definition and calls: stable
- Ruby string operations (PSRAM-heavy): stable
- Thousands of DVI.set_pixel/fill_rect per frame: stable
- Continuous operation for 10+ minutes: no signal loss
- FIFO empty count: 0
