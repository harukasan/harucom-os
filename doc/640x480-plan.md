# 640x480 text mode: status

## Current status: stable (fe=0)

Full-screen CJK stress test (37 rows x 53 wide chars, uniform attrs,
rewritten every frame from mruby) renders at ~2,080 cycles/scanline
with zero FIFO underflows. FIFO-augmented budget is 2,240 cycles,
giving ~160 cycles of margin.

### Render time history

| Milestone | render_last | fifo_empty | Notes |
|-----------|-------------|------------|-------|
| Flash XIP wide font | 20,000+ | severe | QMI contention with PSRAM |
| SRAM wide glyph cache | 2,250-2,350 | 22/3300 | Bus jitter only |
| Bold pre-compute + uniform-attr mixed | **2,076-2,091** | **0** | Current |

### Key optimizations (render path)

- **Bold-adjusted slot at set time**: `put_wide_char_bold` stores
  `slot + WIDE_CACHE_SLOTS` in cell.ch. Eliminates tst/it/addne
  (3 instructions per wide cell) from the render loop.
- **Uniform-attr mixed path**: rows with wide chars and uniform
  attributes skip per-cell attr extraction (ubfx/cmp/bne eliminated,
  3 instructions per cell).
- SRAM wide glyph cache (512 slots, 26 KB regular + bold)
- expanded_nibble[256][2] in SCRATCH_Y (zero DMA bus contention)
- DMA trigger reordering in IRQ handler

### VBlank pre-rendering: attempted and abandoned

VBlank pre-rendering was implemented and tested extensively. The approach
cached dirty rows during VBlank (~360K cycles) and served pre-rendered
scanlines from an SRAM cache. It was abandoned because:

- SRAM instruction fetches from `__not_in_flash_func` code running
  between ISR returns on Core 1 caused bus contention with DMA pixel
  reads during active lines, even when the pre-render itself completed
  within VBlank.
- A flag check (`if (dvi_vblank_pending)`) in the Core 1 main loop
  added ~0.06 fifo_empty/frame from flash instruction fetches + SRAM
  data reads after each ISR return (480 per frame during active lines).
- The pre-render's heavy SRAM traffic during VBlank (~122K cycles for
  4 rows) caused an additional ~0.15 fifo_empty/frame through an
  unknown mechanism (possibly bus arbiter state carryover).
- Net result: pre-rendering worsened FIFO stability vs inline rendering.

The set-time optimization approach (pre-compute metadata at write time
on Core 0, not at VBlank or render time on Core 1) proved effective
with zero Core 1 impact.

### Rejected approaches

**Attr run batching**: pre-scan rows to build same-attr runs. Rejected:
savings (~159 cycles) insufficient alone, SCRATCH_X pressure, race
condition with Core 0 VRAM writes.

**Disabling DVI_DIAGNOSTICS**: saves ~280 cycles but removes
observability. Unnecessary now that render fits within FIFO budget.

## Completed: SRAM wide glyph cache

- 512-slot row-major cache in SRAM (26 KB regular + bold)
- Hash table (1024 entries) maps linear JIS index to cache slot
- Cache populated on Core 0 during `dvi_text_put_wide_char()` (no flash
  access on Core 1)
- Render path uses slot-indexed SRAM ldrh (no multiply, no flash access)
- All nibble mask lookups via expanded_nibble in SCRATCH_Y
- Cache rebuild on overflow scans VRAM and re-maps slots in-place

## Completed: conditional diagnostics

IRQ handler diagnostics (FIFO stat, DWT cycle counting, bus counters) are
now guarded by `#ifdef DVI_DIAGNOSTICS` (defined in CMakeLists.txt).
Removing the definition saves ~200-280 cycles per scanline.

## Completed: 250 MHz stabilization

The sys_clk reduction from 375 MHz to 250 MHz has been completed. The
render now fits within the blanking budget (1,600 cycles) for narrow-only
rows via three optimizations:

- Pre-expanded nibble table (2 KB) in SCRATCH_Y (zero DMA contention)
- Uniform-attribute fast path (skips per-cell attr checks)
- DMA trigger reordering in IRQ handler

Result: render_last=1,763, fifo_empty=0, fifo_min=7 at 250 MHz.
Stable over 3000+ frames at 250 MHz / VREG 1.15V. Standard 60 Hz timing
maintained. See [doc/dvi.md](dvi.md) for details.
