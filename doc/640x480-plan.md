# 640x480 text mode: remaining work

## Full-width font support

Full-width (CJK) rendering is currently unusable with mruby active. The
font bitmap is stored in flash and accessed via XIP during the DMA IRQ
handler. QMI bus contention with mruby's PSRAM access (CS1) stalls flash
reads (CS0), causing render times of 20,000+ cycles (budget: 8,000).

### Candidate approaches

**SRAM glyph cache**: cache the bitmap data of visible full-width glyphs
in SRAM. A screen of 106 x 37 cells with at most ~50% wide characters
needs up to ~1,000 unique glyphs x 26 bytes = ~26 KB. On text VRAM change,
copy the glyph from flash to the SRAM cache. The render reads only from
SRAM.

**Pre-render wide rows**: render scanlines containing wide characters
outside the IRQ context (e.g. during vertical blanking or from core 0).
Store rendered scanline data in a buffer. The IRQ handler uses the
pre-rendered buffer instead of calling the renderer for those lines.
See also [VBlank pre-rendering](#vblank-pre-rendering) below.

**Third DMA channel copy**: render into a temporary main SRAM buffer.
Use a third DMA channel to burst-copy the rendered data to the line buffer
during horizontal blanking. This decouples the CPU render from the HSTX
DMA read timing.

## Further render optimizations

### Current timing budget (250 MHz, 2:1 ratio)

```
Blanking:   160 px x 10 cyc/px = 1,600 cycles
Active:     640 px x 10 cyc/px = 6,400 cycles
VBlank:      45 lines x 8,000  = 360,000 cycles/frame
HSTX FIFO:  8 entries x 40 cyc = 320 cycles buffer

Uniform-attr render:  ~1,484 cycles (pure), ~1,763 with IRQ overhead
Mixed-attr render:    ~1,800 cycles (pure), ~2,050 with IRQ overhead
IRQ overhead:         ~280 cycles (diagnostics, timing, descriptor build)
```

### VBlank pre-rendering

Move scanline rendering from the per-scanline IRQ to the vertical blanking
period (360,000 cycles/frame). Pre-render all dirty rows into a per-row
scanline cache. The IRQ handler points DMA at the cached data instead of
rendering inline.

Scanline cache per text row: 13 scanlines x 640 bytes = 8,320 bytes.
Full screen (37 rows): 307 KB -- too large for SRAM. Practical options:

- **Partial cache**: allocate 3-4 row buffers (~25-33 KB). Typical console
  changes 1-2 rows per frame. LRU or dirty-priority eviction.
- **Ring cache**: keep a sliding window of recently rendered rows. Rows
  scrolled off screen are evicted.

Benefits:
- Eliminates rendering from the IRQ handler entirely for cached rows
- Zero bus contention during active pixels (DMA reads cache, CPU idle)
- Mixed-attr and wide-char rows are no longer timing-critical
- VBlank provides 360,000 contention-free cycles for rendering

Implementation: on VBlank (frame_count increment in IRQ), signal Core 1's
main loop (WFE) to render dirty rows into the cache. The IRQ handler checks
cache validity per text row and falls back to inline rendering on miss.

### Conditional diagnostics

The IRQ handler includes FIFO status reads, DWT cycle counting, and bus
counter polling on every scanline (~280 cycles overhead). These are useful
for development but unnecessary in production.

Options:
- Runtime flag to disable diagnostics (saves ~200-280 cycles/scanline)
- Sample diagnostics every Nth scanline (e.g. every 100th)
- Compile-time `#ifdef DVI_DIAGNOSTICS` guard

With diagnostics disabled, uniform-attr IRQ time drops from ~1,763 to
~1,484 cycles, providing ~116 cycles of blanking margin even without FIFO
buffering.

### Free SCRATCH_Y stack_dummy

The pico-sdk default Core 0 stack (2 KB) occupies SCRATCH_Y but is unused
at runtime (Core 0 uses a 32 KB BSS stack). Freeing it via
`PICO_STACK_SIZE=0` or a custom linker script opens 2 KB for:

- text_palette32 (64 bytes): move attr change lookups off Main SRAM bus
- Additional pre-computed tables for wide font rendering
- Larger expanded_nibble with bold-specific pre-computation

### Mixed-attr path improvement

Mixed-attr rows currently render at ~2,050 cycles (including IRQ overhead),
exceeding the 1,600 cycle blanking budget. The FIFO absorbs the overflow
(fifo_min=1 observed occasionally), but eliminating it would improve
robustness.

**Attr run batching**: scan the row at text-row boundaries (every 13
scanlines) to find contiguous runs of the same attr. Store a compact run
list (attr, start_col, end_col). The render loop processes each run without
per-cell attr checks, approaching uniform-attr speed for rows with few
attr changes (e.g. a prompt with 2-3 color regions).

**H_BACK_PORCH increase**: increase from 48 to 80 pixels for +320 cycles
blanking budget (1,600 to 1,920). Frame rate drops from 60.0 to 57.9 Hz
(barely perceptible). One-line code change as a simple fallback.

## Completed: 250 MHz stabilization

The sys_clk reduction from 375 MHz to 250 MHz has been completed. The
render now fits within the blanking budget (1,600 cycles) via three
optimizations:

- Pre-expanded nibble table (2 KB) in SCRATCH_Y (zero DMA contention)
- Uniform-attribute fast path (skips per-cell attr checks)
- DMA trigger reordering in IRQ handler

Result: render_last=1,763, fifo_empty=0, fifo_min=7 at 250 MHz.
Stable over 3000+ frames at 250 MHz / VREG 1.15V. Standard 60 Hz timing
maintained. See [doc/dvi.md](dvi.md) for details.
