# 640x480 text mode: remaining work

## Full-width font support

Full-width (CJK) rendering is currently unusable with mruby active. The
font bitmap is stored in flash and accessed via XIP during the DMA IRQ
handler. QMI bus contention with mruby's PSRAM access (CS1) stalls flash
reads (CS0), causing render times of 20,000+ cycles (budget: 12,000).

### Candidate approaches

**SRAM glyph cache**: cache the bitmap data of visible full-width glyphs
in SRAM. A screen of 106 x 37 cells with at most ~50% wide characters
needs up to ~1,000 unique glyphs x 26 bytes = ~26 KB. On text VRAM change,
copy the glyph from flash to the SRAM cache. The render reads only from
SRAM.

**Pre-render wide rows**: render scanlines containing wide characters
outside the IRQ context (e.g. during vertical blanking or from core 0).
Store rendered scanline data in a buffer. The IRQ handler uses the
pre-rendered buffer instead of calling the renderer for those lines.

**Third DMA channel copy**: render into a temporary main SRAM buffer.
Use a third DMA channel to burst-copy the rendered data to the line buffer
during horizontal blanking. This decouples the CPU render from the HSTX
DMA read timing.

## Reducing sys_clk from 375 MHz

375 MHz (3:1 ratio) is stable but requires VREG 1.30V and QMI CLKDIV=4.
250 MHz (2:1) is nearly stable (fifo_empty ~1/10 sec) at VREG 1.15V with
no QMI change.

### Why 375 MHz works and 250 MHz does not

The FIFO underflow at 250 MHz is caused by main SRAM bus contention
between DMA pixel reads and CPU render reads during the active pixel
period. The key timing relationship:

```
                        IRQ fires
                           |
  blanking period (160px)  |   active pixel period (640px)
  |<------- sync -------->||<-------- pixel data -------->|
  |                        |                               |
  |  DMA: sync commands    |  DMA: pixel reads from SRAM   |
  |  CPU: rendering        |  CPU: rendering (if not done) |
  |  (no bus contention)   |  (bus contention!)            |
```

At 250 MHz (2:1):
- Blanking = 160 pixels x 10 cyc/px = 1,600 cycles
- Render = ~2,540 cycles
- Render overflows into active period by 2,540 - 1,600 = **940 cycles**
- During these 940 cycles, CPU and DMA compete for main SRAM

At 375 MHz (3:1):
- Blanking = 160 pixels x 15 cyc/px = 2,400 cycles
- Render = ~2,137 cycles
- Render finishes entirely within blanking (**0 overlap**)
- DMA runs with zero CPU bus contention during pixel output

The goal: make the render finish within the blanking period at 250 MHz.
Target render time < 1,600 cycles.

### Approach 1: Reduce render cycles (most promising)

Current render: 106 columns x ~20 cycles/char = ~2,137 cycles at 375 MHz.

The inner loop processes 2 chars per iteration (ldrd pair). Per character:

| Operation | Reads | Writes | ALU | Cycles |
|---|---|---|---|---|
| Load cell (ldrd, amortized) | 0.5 | - | 1 | ~1.5 |
| Attr check (ubfx + cmp) | - | - | 2 | ~2 |
| Font lookup (ldrb narrow_row_cache) | 1 | - | 1 | ~2 |
| Nibble mask lookup (2x ldr nibble_mask) | 2 | - | - | ~4 |
| Pixel computation (and, eor) | - | - | 4 | ~4 |
| Pixel output (2x str, 6px) | - | 2 | 1 | ~3 |
| **Total per char** | **3.5** | **2** | **9** | **~16.5** |

SRAM accesses dominate: 5.5 reads + writes per char, each potentially
stalled by DMA bus contention.

**Option A: Switch to 8px font (80 columns)**

8px font gives 80 columns (640/8) with 2 full 32-bit word outputs per char
(no overlapping writes needed for the 6px remainder). Fewer columns means
fewer iterations and fewer SRAM accesses.

| Metric | 6px (106 cols) | 8px (80 cols) | Reduction |
|---|---|---|---|
| Iterations (ldrd pairs) | 53 | 40 | -25% |
| SRAM reads (cell + font + mask) | ~370 | ~280 | -24% |
| SRAM writes (line_buf) | ~212 | ~160 | -25% |
| Estimated cycles (375 MHz) | ~2,137 | ~1,600 | -25% |

Estimated render at 250 MHz with bus contention: ~2,000 cycles. With
blanking budget of 1,600 cycles, this still overflows by ~400 cycles.
Marginal improvement, not sufficient alone.

**Option B: Uniform-attribute fast path**

Text consoles typically have long runs of the same attribute (e.g. entire
rows of white-on-black). The attr check (ubfx + cmp + bne) costs 3 cycles
per char. If we detect uniform attr before the loop, the inner loop skips
all attr checks.

Savings: 106 chars x 3 cycles = ~318 cycles. This brings 6px render from
~2,137 to ~1,819 cycles, or 8px from ~1,600 to ~1,362 cycles.

**Option C: Move nibble_mask and text_palette32 to SCRATCH_Y**

nibble_mask (64 bytes) and text_palette32 (64 bytes) are currently in main
SRAM, competing with DMA for bus bandwidth. Moving them to SCRATCH_Y
eliminates 4 SRAM reads per char (2 nibble_mask + 2 palette on attr change).

With SCRATCH_Y now empty (line_buf and descriptors moved out), 128 bytes
easily fit. This reduces main SRAM traffic by ~212 reads per scanline
(2 nibble_mask reads x 106 chars), freeing bus bandwidth for DMA.

Estimated impact: ~5-10% render time reduction by avoiding SRAM stalls on
nibble_mask accesses.

**Option D: Skip unchanged rows**

For static text, most rows don't change between frames. Track a dirty flag
per row and only re-render dirty rows. For unchanged rows, reuse the
previous line_buf contents by pointing the DMA descriptor at the same
buffer.

This eliminates rendering entirely for static rows. The text console
typically changes only 1-3 rows per frame (cursor line, status line), so
97%+ of scanlines would be zero-cost.

Complexity: need to invalidate the dirty flag on text_vram writes, and
manage line_buf lifetime across frames (can't reuse if the other buffer
is needed for a dirty row's scanline).

**Option E: Combined approach**

Combining options B + C + 8px font:
- 8px font: ~1,600 cycles base
- Uniform attr: -240 cycles
- SCRATCH_Y LUTs: -100 cycles
- Estimated: ~1,260 cycles at 375 MHz, ~1,500 at 250 MHz

This would fit within the 1,600 cycle blanking budget at 250 MHz with
margin to spare.

### Approach 2: Increase blanking period

Increasing H_back_porch extends the blanking period, giving the render
more cycles before pixel DMA starts.

| H_back_porch | H_total | blanking cycles (250 MHz) | frame rate | render margin |
|---|---|---|---|---|
| 48 (standard) | 800 | 1,600 | 60.0 Hz | -940 (overflow) |
| 88 (sample) | 840 | 1,680 | 56.7 Hz | -860 |
| 128 | 880 | 1,760 | 54.1 Hz | -780 |
| 208 | 960 | 2,080 | 49.6 Hz | -460 |
| 254+ | 1054+ | 2,540+ | 45.1 Hz | 0 (fits) |

To eliminate overlap entirely at current render time (2,540 cycles), the
blanking period must be 254+ pixels, giving H_total = 1054 and frame rate
45.1 Hz. This is too low for most monitors.

However, combined with render optimization (approach 1), less blanking
increase is needed:

| render target | blanking needed | H_back_porch | frame rate |
|---|---|---|---|
| 2,540 (current) | 254 px | 142 | 53.3 Hz |
| 2,000 (8px font) | 200 px | 88 | 56.7 Hz |
| 1,600 (combined opt) | 160 px | 48 | 60.0 Hz (no change) |
| 1,260 (all opts) | 126 px | 48 | 60.0 Hz (fits standard) |

The combined optimization (approach 1E) achieves the target without any
blanking increase, maintaining standard 60 Hz timing.

### Approach 3: Non-integer HSTX divider

Use a fractional clk_hstx divider for intermediate ratios:

| sys_clk | clk_hstx | ratio | blanking cycles | VREG |
|---|---|---|---|---|
| 250 MHz | 125 MHz (/ 2) | 2.0:1 | 1,600 | 1.15V |
| 300 MHz | 125 MHz (/ 2.4) | 2.4:1 | 1,920 | 1.15-1.20V |
| 375 MHz | 125 MHz (/ 3) | 3.0:1 | 2,400 | 1.30V |

300 MHz with fractional divider (2.375 in 8.4 fixed-point gives 126.3 MHz
HSTX, 25.26 MHz pixel clock) provides 1,920 blanking cycles. Combined
with moderate render optimization (option B or C), the render could fit
within blanking at a lower voltage than 375 MHz.

### Recommended path

1. **Short term**: stay at 375 MHz (proven stable, no functional impact)
2. **Medium term**: implement options B (uniform attr) + C (SCRATCH_Y LUTs)
   and measure render reduction. If render drops below 1,600, try 250 MHz.
3. **If needed**: switch to 8px font (80 columns) for further reduction
4. **If still tight**: increase H_back_porch to 88 (matching the sample,
   56.7 Hz) for extra margin
